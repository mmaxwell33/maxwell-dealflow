Developer Handoff · Internal Documentation

DealFlow CRM

System overview, end to end.

A real-estate agent's full working tool: lead intake, viewings, offers, transaction pipeline, stakeholder portals, automated emails, push notifications, and a personal money brief — every screen, every email, every automation, mapped from first touch to keys-in-hand.

Stack at a glance

Frontend

Vanilla-JS Progressive Web App. Installs to the home screen. Works offline (cached shell). Web-Push subscribed device.

Backend

Supabase — PostgreSQL with Row-Level Security, Edge Functions (Deno), Auth, public Storage buckets, pg_cron + pg_net for scheduled HTTP.

Hosting & CDN

Vercel auto-deploys from the master branch. Global edge cache. Service worker handles offline + cache-busting.

External APIs

Gmail (OAuth refresh token), VAPID Web Push, OpenAI (LLM + TTS), Bank of Canada Valet (rates), Tavily (live news).

DealFlow · Stakeholders

Who's involved in a deal

Generic role names. Each role gets different views, different emails, and different access tokens.

Agent (sole authenticated user)
Clients (token-based portal access)
Third parties (email-only)
Pre-client (no record yet)

Access model

  • Agent — authenticated via Supabase Auth (email + password). Sees everything for their own clients only (RLS by agent_id).
  • Buyer / Seller / Co-Buyer — no login. Receive a unique token in each email. Token unlocks a private portal page (respond.html, portal.html) for one deal only.
  • Listing Agent / Lender / Lawyer / Inspector — no app access. Receive emails only with attached PDFs and contact details. Cannot log in.
  • Brokerage — no app access. Receives commission breakdown emails only.
  • Lead — no client record yet. Submits an intake form via an open public URL. Becomes a Buyer or Seller record once Agent qualifies them.

DealFlow · Architecture

How the pieces fit together

Key architectural choices

  • Single-agent app, multi-agent capable. RLS by agent_id everywhere. The "Agent Portal" screen lets the primary user provision additional agents — each gets their own siloed view of clients, deals, and data.
  • No client login. Clients receive single-use VAPID-signed tokens in every email. Tokens are stored on the row and rotated when the deal advances.
  • Edge Functions for anything sensitive. Service-role keys never touch the browser. The frontend calls anonymous endpoints with the user's JWT; functions do the privileged work.
  • pg_cron for scheduled work. Cron jobs in PostgreSQL fire HTTP POSTs to the function URLs every morning, on a fixed UTC schedule. No external scheduler.

RLS pattern

Every "owned" table has the same pair of policies:

  • SELECT using (agent_id = auth.uid()) — agent can read only their own data
  • INSERT/UPDATE/DELETE with check (agent_id = auth.uid()) — agent can only mutate their own rows
  • Token-gated tables (viewing_responses, client_reviews) use SECURITY DEFINER functions that match the supplied token against the row's stored token before returning data.

DealFlow · Phase 4

The pipeline — accepted to closed

Active deal

Auto-advance triggers

  • Inspection complete + condition checked off → Accepted → Under Contract. (Skipped if "no inspection" flagged.)
  • Financing approval email logged + checklist closed → Under Contract → Conditions met.
  • Walkthrough scheduled within 48h of close date → Conditions → Walkthrough. (Skipped if "no walkthrough" flagged.)
  • Close date reached + walkthrough confirmed → Walkthrough → Closing.
  • Agent marks "keys delivered" → Closing → Closed. Commission row finalized. Review request queued.

Stakeholder portal (portal.html)

Each non-agent stakeholder gets a private portal link. Token-gated, no login. Shows the segmented progress bar (sums to 100%), per-stage tasks, role-filtered documents. Skipped stages render as "No inspection" / "No walkthrough" instead of being hidden. The same renderer is used inside the agent app for consistency.

Per-stage tasks (pipeline_tasks)

Buyer-side checklist

  • Wire deposit (5 days)
  • Schedule home inspection
  • Submit mortgage docs
  • Order home insurance
  • Final walkthrough
  • Pay closing costs

Seller-side checklist

  • Confirm closing date with lawyer
  • Cancel utilities (effective close)
  • Schedule moving
  • Final walkthrough access
  • Sign closing documents
  • Hand over keys

Re-engagement (stale deal nudges)

When a client has had no activity (no viewing booked, no email sent, no offer touched) for 7+ days, daily-automation queues one of four template emails through the Approval Queue (so the agent reviews before send):

  • reengagement_brief — generic re-touch ("Just checking in — anything new on your search?")
  • reengagement_buyer_match — for buyers, mentions a recent matching listing
  • reengagement_pre_approval — for buyers without a noted pre-approval, suggests connecting with a lender
  • reengagement_seller — for sellers, mentions market activity in their neighbourhood

Each candidate logs a REENGAGEMENT_QUEUED activity entry to prevent duplicate queueing.

Approval Queue

All system-generated emails flow through approval_queue. The agent sees them on the Approvals screen, can edit / send / discard. Some emails are flagged "auto-approve" (intake confirmations, viewing reminders) — those skip the queue and call Approvals.approve() immediately on insert.

DealFlow · Money Briefing

Daily personal-finance brief

New feature

A separate workflow that generates a daily Canadian personal-finance briefing for the agent personally — not for clients. Audio + written newsletter, delivered to the agent's Gmail and the in-app Money Brief tab every morning at ~6:30 AM.

What's in the briefing

  • Snapshot: live BoC rate, XEQT close, NL avg home price, CPI
  • Mortgage rates: 5-yr fixed, variable, sample $400K monthly payment
  • Three stories: today's top Canadian finance + St. John's local news (sourced from Tavily) with "Plain English" callouts
  • Personal plan: emergency fund target, trading platform comparison, $9K-on-May-17 + $16K-in-October distribution tables, core picks (XEQT, VFV) + satellite risk-on picks, monthly savings deployment based on agent's income vs expenses, tax-strategy estimates, and a daily wealth-wisdom quote
  • Watch list: dates coming up that matter
  • Audio: 9-10 minute 2-host podcast (Avery + Sam) reading the brief in conversational style

In-app tab supports collapsible sections + a sticky audio player at the top. 30-day archive of past briefings is browsable.

DealFlow · Push & Privacy

Notifications, push, privacy

Web Push (VAPID)

Each device subscribes during onboarding using VAPID P-256 / JWK keys. Subscription stored in push_subscriptions. Sending is done by the send-push edge function — fan-out to every active device of the recipient agent.

Privacy & data masking

  • Click-to-reveal — Names, phones, emails appear masked in lists. One tap reveals (logs to activity_log for audit).
  • Token-gated portals — Buyers, Sellers, and stakeholders never log in. Each receives a single-use VAPID-signed token in their email. Tokens are stored as hashes server-side; raw tokens never logged.
  • Token rotation — Portal tokens rotate when a pipeline row advances stages. Old tokens are invalidated.
  • Role-filtered documents — The portal renders different document subsets per stakeholder role. Lawyer sees the agreement + addenda; Inspector sees the property card + access notes only; Buyer sees everything.
  • RLS by agent_id — Every owned table has the same policy: USING (agent_id = auth.uid()). Cross-agent leakage impossible.
  • No client-side service-role keys — All sensitive operations (commission writes, agent provisioning, push fan-out) run inside Edge Functions with the service-role key. The browser only ever holds the user's own JWT.
  • Auto-prune expired push subscriptions — App pulls 410/404 subscriptions on send to prevent push spam to dead devices.
