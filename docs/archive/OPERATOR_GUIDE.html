01What Maxwell DealFlow Is

Maxwell DealFlow is a Progressive Web App (PWA) — it runs in a browser but installs to your iPhone home screen and works offline like a native app. It is the operating system for one real-estate agent's brokerage: tracking clients, booking viewings, sending offers, managing closings, talking to builders, and emailing everyone involved without ever leaving the app.

Three things separate it from an off-the-shelf CRM:

  1. It speaks to clients directly. Every viewing booking, build update, and stakeholder portal lives at its own public URL. No login required for the recipient — they tap the link in their text or email and the page just works.
  2. It sends real emails from Maxwell's real Gmail. Through a Supabase Edge Function that signs into Gmail's API, every confirmation, reminder, and update reaches the inbox from maxwell@…, not a no-reply address. Every outbound email signs off as eXp Realty.
  3. It pushes notifications to the phone even when closed. Web Push (VAPID, P-256 / JWK) wakes the phone the moment a client responds, an offer comes in, or a viewing is confirmed.

02Sign In

Open the app URL on iPhone or laptop. The first screen is a login form. Email + password logs you into Supabase Auth. After login, the app loads index.html, the home dashboard. Use "Add to Home Screen" to install the PWA as a real app icon.

If a session expires, Supabase refreshes the token automatically. If refresh fails, you're sent back to the login screen. To reset a password, run supabase auth users update <email> --password <new> from a machine with the Supabase CLI signed in.

03The Dashboard (Overview)

First screen after login. It answers one question: what needs my attention right now?

Top cards show: active clients, viewings today/this week, pipeline value (sum of offer_amount across live deals), outstanding tasks across every deal, and pending approvals. Below the cards, an activity feed lists every recent action — sourced from activity_log. Tap any item to jump to its source screen.

The bell icon (top right) opens the in-app notification tray. The moon toggles dark/light mode. The purple "Ask AI" button opens the assistant (Section 17).

04Clients

Where: Clients tab. File: js/clients.js. Table: clients.

The master list of every buyer or seller. Each row shows name, last contact date, and quick-action buttons.

Privacy masking: names and contact info are masked by default — J••• A••• instead of Jane Anderson. Tap to reveal. Powered by App.privateName() and App.privateContact().

Add a client: tap +. Fill name, email, phone, status (Active / Archived / Lead), notes. Save.

Intake form: every client gets a public intake link (intake.html?token=…). Send it; they fill it; their answers land in intake_responses and a push fires on Maxwell's phone.

Responses tab: intake submissions, stakeholder replies, viewing feedback all show up here, sourced from client_responses and form_responses.

05Viewings (Showings)

Where: Viewings tab. File: js/viewings.js. Tables: viewings, viewing_responses.

Book a viewing: tap +, pick client, enter address, MLS, date, time, duration. Save. Three things happen automatically:

  1. A row inserts into viewings.
  2. An email is queued (Section 11) with a calendar invite (.ics) attached.
  3. A respond link is generated and embedded in the email — when the client taps it, they land on respond.html?token=… (no login) and can confirm, reschedule, or cancel.

Auto-completion: after a viewing's scheduled end-time passes, it auto-flips to "completed" and prompts Maxwell with the Agent Feedback Modal — log how it went, the client's reaction, follow-up notes, and a 1–5 score. Goes back into the client's history.

06Offers

Where: Offers tab. File: js/offers.js. Tables: offers, pending_offers.

Create an offer: pick a client, enter property address, offer amount, conditions, deadlines (financing, inspection, closing). Save. The system formats it into a presentable PDF using the built-in PDF export.

Pending offers sit in pending_offers for editing. Once sent, they move into offers. Accepted offers auto-promote to the Pipeline (Section 7). The Requests tab shows change requests sent in by clients or stakeholders through their portals.

07Pipeline (the deal tracker)

Where: Pipeline tab. Files: js/offers.js, js/tracker.js. Tables: pipeline, pipeline_tasks, checklist_items.

Once an offer is accepted, it lives here until closing. Stages: Acceptance → Financing → Inspection → Walkthrough → Closing. Each has a deadline.

Transaction Room: tap any deal to open its detail view — property details, all five stage countdowns, a checklist of tasks, document area for contracts and inspection reports, and an activity timeline scoped to that deal.

Daily reminders: the daily-automation function (Section 19) checks for upcoming deadlines and pushes a notification: "Inspection deadline for 100 Sample Build Lane in 2 days."

PDF export dumps the entire deal for handoff to a lawyer or lender. Archive a closed deal — stamped with archived_at and kept for commission tracking.

08New Builds

Where: New Builds tab. Files: js/newbuilds.js, js/extras.js. Table: new_builds.

When a client buys a not-yet-built home, that's a new build — typically 12–18 months of construction. Each record tracks client info, builder info (builder_name, builder_email, community, lot_address), current stage (1–9 milestones), estimated completion, and MLS number.

Builder Portal: each build has a public URL (builder.html?token=…) the builder visits to upload photos and tick milestones — no login required. Production version: builder-portal.js.

Client view: same data, cleaned up for clients at portal.html?token=….

Builder visit loop: when a build hits a milestone needing inspection, the system emails the builder a calendar invite for a site visit (migration 019).

09Commissions

Where: Commissions tab. File: js/extras.js. Table: commissions.

Tracks earnings on closed deals. Auto-calculates as sale price × commission rate (default 2.5%). Records co-listing splits when applicable. Shows a running YTD total broken out by month.

10Reports

Where: Reports tab. File: js/extras.js. Table: reports + aggregations.

Pre-built monthly review reports: month-over-month volume (viewings booked, offers written, deals closed), conversion funnel (leads → showings → offers → closings), client lifetime value, and source attribution. Tap any report to drill in or export to PDF.

11Email (Inbox + Send)

Where: Email tab. Files: js/extras.js, js/notifications.js. Edge function: send-email. Tables: email_inbox, approval_queue.

Two-way Gmail integration. Outbound: send-email authenticates against Gmail's API using stored OAuth and sends from Maxwell's real account. Inbound: fetch-inbox polls Gmail every few minutes and writes new messages to email_inbox, keyed by gmail_thread_id so replies thread correctly.

Templates in js/notifications.js cover viewing confirmation, viewing reminder, offer sent, offer accepted, build update, closing congratulations, etc. Each has placeholders filled at send time.

Approvals queue: by default every system email lands in approval_queue first. Maxwell reviews on the Approvals screen and taps Approve. Some templates have auto-approve enabled and skip the queue. Migration 017 enforces a per-recipient rate limit so the system can't spam.

12Approvals

Where: Approvals tab. File: js/extras.js. Table: approval_queue.

The safety net. Every system-generated email passes through here unless auto-approve is set. Each row shows From / To / Subject and a body preview. Buttons: Approve (sends now), Edit (modify before sending), Reject (discard).

13Broadcast

Where: Broadcast tab. File: js/extras.js.

Send a single message to a filtered list of clients at once. Useful for: open-house announcements, market updates, brokerage news. Choose a filter (active / past / leads), write the message, preview, send. Each individual email goes through the approval queue.

14Calendar

Where: Calendar tab. File: js/calendar.js.

A unified calendar showing every dated event: viewings, offer deadlines, pipeline deadlines, builder visits, closings. Tap any event to jump to its source. All dates formatted via App.fmtDate() which handles both plain YYYY-MM-DD and full Supabase timestamps.

15Analytics + Portal Traffic

Where: Analytics and Portal Traffic tabs. Files: js/analytics.js, js/portal-traffic.js. Charting: Chart.js. Tables: portal_views + aggregations.

Analytics: high-level dashboard — bar/line charts of viewings per month, offers per month, conversion ratios, sources of business.

Portal Traffic answers: did the link I sent actually get opened? Every portal open inserts into portal_views. The screen shows:

  • Stat cards — total views, today's views, unique clients, top portal type.
  • Views over time chart — 7/30/90-day line chart split by portal type, each in its own color (Client orange, Builder purple, Stakeholder green).
  • By Viewer table — one row per (deal × recipient). Sample Builder Co. opening Client B's build link shows up as its own row, separate from Client B's client portal views. Each row carries a role badge (Client / Builder / Lawyer / Lender) and a subline with deal owner + address.
  • Recent Portal Views — last 20 opens, color-coded pills, recipient name, deal context, plus a "Mark as self" button to exclude self-tests.
This is the screen that confirms yes, the builder actually clicked the link I sent.

16Activity Log

Where: Activity tab. Table: activity_log.

Every meaningful action gets logged: client created, viewing booked, offer sent, email approved, milestone ticked, portal opened. Reverse-chronological feed. Each entry stores actor, verb, object ID, timestamp. The audit trail.

17AI Assistant ("Ask AI")

Where: purple Ask AI button (top right of every screen). File: js/ai.js. Edge function: claude-chat. Model: Claude Haiku.

A chat panel slides in. Ask anything: "Who's closing this month?" "Draft an email to Sample Builder about the framing inspection." "What's my YTD commission?" The assistant has read access to a curated database subset (clients, recent viewings, active builds, open pipeline deals) and feeds that context to Claude before answering. It can draft emails and summarize deals on demand. It's read-only — it cannot send emails or edit data.

18Push Notifications

Files: sw.js, js/app.js. Edge function: send-push. Table: push_subscriptions.

On first install the app asks "Allow notifications?" Tap Allow. The browser hands back a VAPID subscription (P-256 / JWK) which gets stored in push_subscriptions keyed to Maxwell's user ID. From then on, anytime the system needs to alert Maxwell — viewing response, new offer, builder portal open, morning briefing — send-push reads the stored subscription, signs a VAPID JWT, and pushes through Apple's / Google's push servers. Notifications arrive even when the app is closed.

20System Health

Where: Settings → System Health. File: js/extras.js.

Diagnostic screen showing: last successful Gmail sync, last push notification, edge function health (latency per function), push subscription count, database row counts per table, cron last-run timestamps. If anything's red, most likely a Supabase outage or expired Gmail OAuth — fix the latter at Settings → Re-authenticate Gmail.

24Configuration & Deployment

Front-end: GitHub master branch auto-deploys to Vercel. Push to master and within ~30 seconds the new version is live globally. No build step — vanilla HTML/JS/CSS served directly.

config.js holds three values: SUPABASE_URL, SUPABASE_ANON_KEY (public, RLS-protected), VAPID_PUBLIC_KEY.

Service Worker (sw.js) handles offline cache and Web Push delivery. Cache-busts on each deploy via a version string.

Edge Functions (TypeScript / Deno) deploy via the Supabase CLI:

supabase functions deploy send-email
supabase functions deploy send-push
supabase functions deploy claude-chat
supabase functions deploy fetch-inbox
supabase functions deploy daily-automation
supabase functions deploy morning-briefing
supabase functions deploy check-followups
supabase functions deploy backup-data

Each function reads its secrets from Supabase's secret store (Gmail OAuth tokens, VAPID private key, Anthropic API key).

Migrations in supabase/migrations/ apply with supabase db push. Numbered 001_… through 026_… and must be applied in order on a fresh database.

25The Five Surgical Update Rules

From CLAUDE.md. Anyone editing this app — human or AI — follows these:

  1. Read before editing. Use grep or read the file first; never blind-edit.
  2. One function at a time. Change only the specific block. Leave everything else untouched.
  3. Show the diff first. Non-trivial changes show BEFORE / AFTER before applying.
  4. Web-search before implementing new features. Check current best practices and standard APIs.
  5. Verify before committing. Run node --check or grep the changed file before staging.
git add <only the specific files changed>
git commit -m "fix/feat/chore: <what changed and why>"
git push origin master

26Common Operating Procedures

A new buyer client comes in

  1. Clients tab → + → fill name, email, phone → Save.
  2. Tap their card → Send intake form → email goes through approvals.
  3. They submit; you get a push.
  4. Their answers populate the Responses tab on their profile.

Booking a showing

  1. Viewings → + → pick client, address, date, time.
  2. Save. Confirmation email + .ics goes out (after approval if not auto-approved).
  3. Client taps respond link, confirms.
  4. Push fires.
  5. After the showing, the auto-completion modal asks for feedback.

Writing an offer

  1. Offers → + → pick client, fill property, amount, conditions.
  2. Save → PDF auto-generated.
  3. Send to client / co-agent through approvals.
  4. On acceptance, mark accepted → auto-promotes to Pipeline.

Tracking a new build

  1. New Builds → + → pick client, fill builder name + email, lot address, MLS, est completion.
  2. Save → builder gets emailed a portal link automatically.
  3. Builder logs progress at builder.html?token=….
  4. Client sees the cleaner version at portal.html?token=….
  5. Maxwell watches both via the Portal Traffic screen.

Closing a deal

  1. Pipeline → tap deal → tick remaining checklist items.
  2. Set closing date completed.
  3. Deal auto-flips to Archived → appears in Commissions YTD total.
  4. Closing congratulations email auto-fires.

28The 30-second pitch

Maxwell DealFlow is a one-person brokerage's entire operating system in a single PWA. It tracks every client from lead to closing, sends every email and notification automatically, gives the builder and the lawyer their own portal, and runs on a phone in offline mode. It deploys with a single git push. It costs roughly $20/month to run (Supabase + Vercel + Anthropic API). It gives a solo agent the operational leverage of a small team.

That's the whole system.
