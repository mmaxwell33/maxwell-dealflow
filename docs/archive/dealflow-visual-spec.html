Developer Handoff · Visual Specification

DealFlow CRM

Every screen. Every email. Drawn.

Visual mockups of the agent's app, public stakeholder pages, and every transactional email — drawn as wireframes a developer can implement against. Generic role names throughout (Agent, Buyer, Seller, Lawyer, Lender, Inspector, Brokerage, Co-Buyer, Lead).

How to use this document

  • Each screen is shown as a phone-frame mockup (DealFlow is a mobile-first PWA).
  • Each email is shown as a Gmail-style frame: subject line, sender, body layout, CTA button, footer.
  • Triggers and recipients are noted under each mockup.
  • Color coding: screen agent app screen · public page token-gated public page · email outbound email.

A · Agent screens

Login

screen

Purpose

Single agent (or team) authentication. Email + password via Supabase Auth.

Key elements

  • Email + password fields
  • "Sign in" button (primary)
  • "Forgot password?" link → Supabase reset flow
  • App version in footer

Behavior

  • On success: redirects to Overview screen
  • JWT stored in localStorage via Supabase JS SDK
  • Service worker + Web Push registration prompted after first login

A · Agent screens

Pipeline

screen

Purpose

Every deal in motion, from acceptance to close. Same renderer used for the client portal.

Per-deal card shows

  • Property address + buyer/seller (masked) + close date
  • Current stage label + percentage
  • Single segmented progress bar (5 segments summing to 100%)
  • Skipped stages render as "No inspection" / "No walkthrough" — never hidden
  • Task count: "3 of 6 tasks done"
  • Closed deals: green border, commission shown, "Request Review" button

Auto-advance triggers

  • Inspection complete → Accepted → Under Contract
  • Financing approved → Under Contract → Conditions met
  • Walkthrough scheduled → Conditions → Walkthrough
  • Agent marks "keys delivered" → Walkthrough → Closing → Closed

Approvals

screen

Purpose

Every system-generated outbound email passes through here for the agent to review before sending.

Per-pending email shows

  • Yellow "Awaiting approval" banner with template type
  • To, Subject, body preview (3 lines)
  • Approve · Edit · Discard buttons
  • Queued time

Auto-approve emails

  • Bypass this screen entirely (e.g. intake_confirmation, viewing_booked)
  • Triggered with auto_approve: true on Notify.queue()
  • Still log a row here (marked "Auto") for audit history

screen

Two-way Gmail conversations grouped by thread. Synced via Gmail API.

Per-thread card

B · Public pages

intake.html — Lead intake form

public page

Purpose

Open public form. No login. Lead's first touch.

Fields

  • Name, email, phone
  • Buy / Sell / Both segmented control
  • Price range, timeline
  • Free-text notes

On submit

  • Inserts row in intake_responses via anonymous public RPC
  • Fires Web Push to agent: "New lead"
  • Auto-sends confirmation email to lead from agent's Gmail
  • Shows thank-you page: "I'll be in touch within 24h."

URL

https://<app>.vercel.app/intake.html?ref=qr — optional ?ref for source tracking

B · Public pages

respond.html — Viewing response

public page

Purpose

Token-gated viewing-response page. No login. Token in URL.

Fields shown

  • Personal greeting (first name only — pulled from token's row)
  • Property hero image, address, time, duration, access notes
  • Three big tappable buttons

Behavior

  • "Confirm" → SECURITY DEFINER RPC writes to viewing_responses, fires push to agent
  • "Suggest different time" → opens date/time picker, agent reviews on Responses screen
  • "Can't make it" → marks declined, agent gets push, viewing row archived
  • Token is single-use after response — page becomes read-only confirmation

URL

https://<app>/respond.html?t=<token>

portal.html — Stakeholder deal portal

public page

Purpose

Live deal-progress page for Buyer / Seller / Stakeholder. Token-gated, no login.

Sections

  • Header — property + days-to-close
  • Segmented 5-stage progress bar (sums to 100%, current stage highlighted)
  • "What's next" — current stage + the immediate next action with date

  • SECURITY DEFINER RPC returns role-appropriate doc list
  • Same renderer as agent's Pipeline screen for consistency

review.html — Post-close review

public page

Purpose

Single-use review form sent post-close. Token-gated. Star ratings + free-text.

Five questions

  • Communication (1–5 stars)
  • Negotiation (1–5 stars)

C · Emails

Email — Re-engagement (stale buyer)

approval

When it sends

Daily-automation cron (5:30 AM NDT) detects clients with no activity in 7+ days. Queues one of four template variants based on client state.

Four variants

  • reengagement_brief — generic check-in
  • reengagement_buyer_match — buyer + matching listing exists
  • reengagement_pre_approval — buyer w/o noted pre-approval
  • reengagement_seller — seller-side, mentions market activity

Approval flow

  • Goes through approval_queue — agent reviews + edits before send
  • Idempotency: REENGAGEMENT_QUEUED activity entry prevents duplicates
  • Reset after agent sends or client replies

Summary

What this document covers

A visual reference for developers building DealFlow CRM. Every screen mocked. Every transactional email shown. All names generic.

Section A · Agent app screens

  • Login
  • Overview / Dashboard
  • Clients (list + detail)
  • Viewings + calendar
  • Offers (draft, dispatch, accept)
  • Pipeline (deal stages)
  • Approvals (email queue)
  • Inbox (Gmail-synced)

Additional screens follow same patterns: Tracker, Form Responses, Calendar, Commissions, Reports, Analytics, Reviews, Settings, Agent Portal, System Tools, Email Send, Broadcast.

Section B · Public token-gated pages

  • intake.html — lead enquiry form
  • respond.html — viewing confirm/decline
  • portal.html — stakeholder deal portal
  • review.html — post-close review

Section C · Transactional emails

  • Intake confirmation (auto)
  • Viewing booked (auto)
  • Viewing reminder 24h (cron)
  • Viewing follow-up (approval)
  • Offer dispatched (auto)
  • Offer accepted — Buyer / Lawyer / Lender / Inspector / Brokerage
  • Conditions cleared (auto)
  • Walkthrough booked (auto)
  • Closing day reminder 24h (cron)
  • Deal closed — Buyer / Seller / Brokerage
  • Review request (manual)
  • Re-engagement × 4 variants (approval)

Notes for the implementation team

  • Color palette — primary #1a1d2e (navy), accent #5b5bd6 (indigo), success #1a8a4f, warning #b8870c, danger #d04a3c, cream bg #fefdf9, soft band #f5efdf
  • Type — system sans-serif everywhere; Georgia / Times serif for headlines + key numbers
  • Mobile-first — all screens designed for phone viewport. Bottom nav (Home, Clients, +, Approvals, Inbox) is primary navigation. Sidebar appears on tablet/desktop.
  • Privacy — masking by default on all client lists. Click-to-reveal logs to activity_log.
  • Token-gated pages — never query Supabase with the user's JWT. Use SECURITY DEFINER RPCs that take the token + check it server-side.
  • Service worker — versioned cache (v70+). Bumps on every deploy. Network-first for HTML/JS/CSS, cache-first for icons.
