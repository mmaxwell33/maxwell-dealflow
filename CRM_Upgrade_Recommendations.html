1 Safe-Update Skill Protocol — "Surgical Update Mode"

The Problem We're Solving

Every time a new feature was requested, the old approach was to regenerate large sections of code — which caused working features to break. The new approach uses what we call Surgical Update Mode: a set of strict rules Claude follows when making any change to this codebase.

The 5 Rules of Surgical Update Mode

RULE 1
Never rewrite — only insert or patch. Every change targets a specific named function, section comment, or element ID. Claude identifies the exact insertion point (e.g. "after the /* ── TOP BAR ── */ comment") and adds/changes only those lines.
RULE 2
Changes are shown as diffs first. Before any edit is applied, a before/after preview is shown. You approve. Then it's applied.
RULE 3
One file, one feature, one commit. Each update only touches the minimal files needed. If adding a profile dropdown, only the HTML and CSS are touched — JS is separate.
RULE 4
CSS uses the existing variable system. No hardcoded colors — every new style uses the :root CSS variables already defined. This means a theme change updates every element at once.
RULE 5
Web research before implementation. Before adding a new feature, Claude searches for the latest, most stable technique for that feature and uses that approach — not an outdated method.

How to Trigger This Mode Every Time

Start every future CRM update request with this phrase:

"Using Surgical Update Mode for Maxwell DealFlow CRM: [describe what you want]. Do NOT rewrite any working sections. Show me the exact lines being changed before applying. Search for the latest best practice before implementing."

3 Apple Liquid Glass Effect — What It Is & Where to Use It

What Is It?

Apple announced Liquid Glass at WWDC 2025 — it's a new design language introduced in iOS 26, macOS Tahoe, and visionOS 26. It's Apple's biggest UI overhaul in 10 years. The effect creates a translucent, frosted-glass panel that dynamically bends and refracts the background behind it — like looking through real glass.

In CSS, the core technique is already available — it's called glassmorphism, using backdrop-filter: blur(). Apple's version is more advanced (using real-time light refraction) but we can achieve 90% of the visual feel with pure CSS.

Glass CSS — The Recipe

/* Liquid Glass Card */ .glass-card { background: rgba(26, 34, 53, 0.55); backdrop-filter: blur(18px) saturate(180%); -webkit-backdrop-filter: blur(18px) saturate(180%); border: 1px solid rgba(96, 165, 250, 0.18); border-radius: 16px; box-shadow: 0 8px 32px rgba(0,0,0,0.4), inset 0 1px 0 rgba(255,255,255,0.08); }

7 Analytics Bug Diagnosis — Why Charts Are Empty

Root Cause Analysis

After reviewing your analytics.js code, I found 3 specific bugs causing the empty charts. No data is lost — it's a data-field naming mismatch and empty-state handling issue.

📅
BUG 1 — "Viewings Over Time" is Empty
The chart only reads v.viewing_date. But when viewings were entered, the field may be stored as date or created_at in your Supabase table. Since no records have viewing_date populated, the dates array is empty → blank chart.
FIX: Change line 147 in analytics.js from if (!v.viewing_date) return; to const raw = v.viewing_date || v.date || v.created_at; if (!raw) return;
👤
BUG 2 — "Viewings Per Client" is Empty
The chart reads v.client_name from viewings records. If your viewings were booked without a client_name field (or if the field is called name or linked via client_id), every row is filtered out and the chart shows empty.
FIX: Add fallback: const n = v.client_name || v.name || v.client || ''; — and ensure Viewings.openAdd() always saves client_name when booking.
🥧
BUG 3 — "Viewing Status" Shows No Graph
The chart reads v.viewing_status || v.status. If your viewings were saved without a status field, statusMap is empty. The code shows "No viewing data yet" text but leaves the canvas behind it — which causes a visual glitch (text over blank canvas).
FIX: When statusMap is empty, hide the canvas entirely with ctx.style.display='none' before showing the message.

✅ The Good News

Your Pipeline Breakdown, Stage Distribution, and Budget Distribution charts work because they read from clients data (which has consistent fields). The viewings-based charts fail because viewings records have inconsistent field names. These are 3 targeted one-line fixes — no rewrite needed.
